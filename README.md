# Best Practices

## Назначение

База знаний с лучшими практиками, извлечёнными из завершённых проектов.
Проекты приходят и уходят — уроки остаются здесь.

## Структура

| Раздел | Назначение |
|---|---|
| `practices/1c/` | Практики разработки для 1С (модули, типовые ошибки, чек-листы) |
| `practices/web/` | Практики веб-разработки |
| `practices/common/` | Общие практики, не привязанные к стеку |
| `practices/tools/` | Инструменты и MCP-серверы: для чего, вердикт tech-radar |
| `practices/anti-patterns/` | Чего избегать и почему |
| `practices/prompts/` | Удачные промпты и skill-паттерны |
| `practices/snippets/` | Переиспользуемые сниппеты и конфиги (без секретов) |

## Как пользоваться

Новый контрибьютор начинает с
[[docs/tutorials/FIRST_CONTRIBUTION|первого кандидата за 10 минут]]. Полная
операционная последовательность описана в
[[docs/how-to/HARVEST_REVIEW_APPLY|harvest-review-apply]].

1. **Начало нового проекта** — открой релевантный раздел `practices/` или
   попроси агента: «прочитай практики по 1С из Best Practices». Автоматически
   это делает skill `apply-best-practices` (запуск из целевого проекта).
2. **Завершение проекта** — перед удалением попроси агента собрать уроки:
   skill `harvest-practice-candidates` заведёт файлы-кандидаты в [[candidates/README|candidates/]],
   а `review-practice-candidates` примет их в `practices/<раздел>/`.
3. **Командная работа** — кандидаты присылаются через pull request (один PR —
   один файл в `candidates/`), `main` под branch protection; см. [[CONTRIBUTING]].

## Конвейер практик

Три явно разделённых этапа (каждый — отдельный skill):

| Этап | Skill | Где запускать | Что делает |
|---|---|---|---|
| Импорт | `harvest-practice-candidates` | из этой базы | Сканирует соседние проекты, заводит файлы-кандидаты в [[candidates/README\|candidates/]] |
| Оценка и принятие | `review-practice-candidates` | из этой базы | Оценивает кандидатов и переносит принятые в `practices/<раздел>/` |
| Доставка | `apply-best-practices` | из целевого проекта | Pull: читает `practices/<стек>` + `common` и вносит применимые уроки в проект со ссылкой на источник |

Мета- и процессные уроки (правила агента, шаблоны, workflow) сюда не попадают —
их маршрут ведёт в стандарт `new-project-rules` (`Правила для нового проекта`)
через его `harvest-project-lessons` / `promote-project-knowledge`.

## Формат записи практики

Каждая практика — отдельный `PC-*.md` с контекстом, проблемой, решением,
применимостью, evidence и происхождением. Контракт: [[docs/reference/PRACTICE_SCHEMA]].

Поиск по section/status/tag/text выполняет `make catalog` или
`scripts/search_practices.py`; справочник: [[docs/reference/PRACTICE_CATALOG]].

## Происхождение (provenance) — обязательно

Каждая запись — практика, инструмент, MCP-сервер, анти-паттерн, промпт, сниппет
и файл-кандидат в `candidates/` — несёт источник:

- **Источник** — проект, из которого пришло.
- **От кого** — кто предложил/подтвердил: пользователь (имя/роль), автор коммита
  или агент, если добавлено автоматически.
- **Дата** и **evidence** (commit/PR/defect) — где есть.

Правило зафиксировано в [[docs/architecture/decisions/ADR-0001-provenance-required]]
и удерживается скилами конвейера. Секреты в происхождении не хранятся — только
имена проектов и людей/ролей.

## Правила

- Эта папка является git-root проекта и находится внутри родительского
  Obsidian vault.
- Markdown редактируется напрямую, без API, helper и второй копии.
- Машиночитаемые спецификации остаются первичными источниками истины.
