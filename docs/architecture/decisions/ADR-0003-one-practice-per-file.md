# ADR-0003. Одна принятая практика хранится в одном файле

- **Статус:** accepted (2026-07-05)

## Context

Кандидаты уже хранятся по модели «один файл — один объект», чтобы независимые
PR не изменяли общий агрегат. Но review-workflow предписывал дописывать принятые
практики в общий `practices/<stack>/README.md`, возвращая merge-конфликты,
неявное владение и неудобную историю изменений при росте базы.

Практике также нужен собственный жизненный цикл: применимость, уровень
доказательности, владелец, дата повторной проверки, замена и конфликты.

## Decision

1. Принятая практика хранится в
   `practices/<stack>/PC-<year>-<number>-<slug>.md`.
2. Имя и `id` сохраняются от исходного кандидата. Файл кандидата остаётся
   журналом решения и ссылается на файл практики через `target`.
3. `practices/<stack>/README.md` — навигационный индекс, а не контейнер записей.
4. Жизненный цикл практики: `trial`, `accepted`, `deprecated`, `superseded`.
5. Уровни evidence:
   - `E0` — мнение или внешний материал без проектного подтверждения;
   - `E1` — один проект или единичный эксперимент;
   - `E2` — два независимых проекта либо воспроизводимый тест;
   - `E3` — несколько проектов и повторяемая автоматическая проверка.
6. Для статуса практики `accepted` требуется минимум `E2`; для `trial`
   допустим `E1`. Статус кандидата `accepted` означает решение review и может
   создать практику в состоянии `trial`.
7. Каждая практика содержит `owner`, `applies_to`, `does_not_apply_to`,
   `last_verified`, `review_by`, `supersedes` и `conflicts_with`.

## Consequences

- Независимые практики меняются без конфликта в общем README.
- CODEOWNERS и история Git точнее отражают владение объектом.
- Устаревание и замена становятся явными и проверяемыми.
- Индексы нужно поддерживать вручную или генерировать.
- Candidate и practice хранят часть одинакового provenance, но выполняют разные
  функции: журнал решения и применяемый source of truth.
