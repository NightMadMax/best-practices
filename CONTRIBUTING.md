# Как вносить вклад

База собирает практики от многих пользователей. Приём — через **Pull Request**:
предлагать может кто угодно, а `main` защищён, поэтому в курированную базу
попадает только проверенное. Прямой push в `main` не используется.

## Прислать кандидата

1. Форкни репозиторий или заведи ветку `candidate/<кратко>`.
2. Скопируй `candidates/_TEMPLATE.md` в
   `candidates/PC-<год>-<номер>-<slug>.md` (номер — следующий свободный).
3. Заполни frontmatter и тело. **Обязательны** `source` (из какого проекта) и
   `added_by` (от кого) — происхождение по
   [ADR-0001](docs/architecture/decisions/ADR-0001-provenance-required.md).
4. Убедись, что нет секретов, токенов и приватных хостов — только способ
   настройки и плейсхолдеры.
5. Открой PR. Один PR — один кандидат (новый файл), чтобы не было
   merge-конфликтов.

## Что происходит дальше

Мейнтейнер разбирает кандидата через skill `review-practice-candidates`:

- **accept** — практика переносится в `practices/<раздел>/`, во frontmatter
  кандидата ставится `status: accepted` и commit; PR вливается;
- **reject** — `status: rejected` с причиной в теле; файл сохраняется как
  журнал решения.

Наполнять `practices/**` напрямую в PR не нужно — туда пишет только review.

## Роли и доступ

- **Контрибьютор** — предлагает кандидатов через PR, без прав записи в `main`.
- **Мейнтейнер** — ревьюит PR, принимает/отклоняет, ведёт `practices/**`.

## Branch protection (настройка репозитория)

Чтобы модель работала, включи защиту `main` в GitHub
(*Settings → Branches → Add rule*):

- require a pull request before merging;
- require review approval;
- (опционально) restrict who can push to `main`.

Это настройка на стороне GitHub — задаётся владельцем репозитория один раз.
