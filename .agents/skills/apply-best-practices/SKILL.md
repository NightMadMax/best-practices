# Доставка практик в проект (pull)

Этот skill применяется **при работе в целевом проекте**, а не в базе. Он не
устанавливается в целевой проект как команда: агент читает этот файл из соседней
базы `Best Practices` и следует ему, оставаясь в корне целевого репозитория.
Модель — pull: база остаётся единственным source of truth, а проект забирает из
неё только применимое и с атрибуцией. Push-распространение в чужие репозитории
не выполняется.

## Определить контекст целевого проекта

1. Работать в корне целевого git-репозитория (текущий проект).
2. Определить стек по манифестам и конфигурации (например: 1С-выгрузки/EDT →
   `1c`; `package.json`/фронтенд-конфиги → `web`; иначе — только `common`).
3. Если стеков несколько, взять все подходящие разделы плюс `common`.

## Найти базу и прочитать практики

1. Найти базу `Best Practices` среди соседних каталогов (родитель проекта):
   каталог с `README.md`, `INDEX.md` и `practices/`.
2. Прочитать индексы релевантных стеков, затем отдельные
   `practices/<стек>/PC-*.md` и `practices/common/PC-*.md`, а также
   кросс-разделы:
   - `practices/tools/README.md` — какие инструменты/MCP брать (вердикт
     `Adopt`/`Trial`), чего избегать (`Hold`);
   - `practices/anti-patterns/README.md` — чего не делать;
   - `practices/prompts/README.md` и `practices/snippets/README.md` — если
     задача про работу с агентами или переиспользуемые конфиги.
3. По умолчанию читать и доставлять только practice-файлы со
   `status: accepted`. `trial` включать только по явному запросу пользователя;
   `deprecated` и `superseded` не применять. Staging `candidates/` не читать.
4. При использовании `practice_report.py` учитывать поле `sections`: default
   report включает `common`, обнаруженные stack-секции и все cross-cutting
   sections. В неоднозначном multi-stack проекте передать каждый стек через
   повторяемый `--section`; не считать отсутствие auto-detection доказательством
   неприменимости.

## Отобрать и применить

1. Составить чек-лист практик, применимых к текущему проекту, с явной привязкой
   «практика → что проверить/изменить здесь».
2. Для применимых практик вносить конкретные правки в целевой репозиторий
   (код, конфигурация, документация), а не абстрактные советы.
3. В каждой правке ссылаться на **ID, отдельный файл и происхождение** практики,
   например `PC-2026-001` из
   `Best Practices/practices/1c/PC-2026-001-<slug>.md` (источник: проект и
   автор из frontmatter). Не создавать кросс-репо wikilinks.
4. Не переносить нерелевантное и не тянуть практику «на всякий случай».

## Точка интеграции

- При бутстрапе нового проекта этот skill может вызываться из
  `create-new-project` стандарта. В рамках доставки сам стандарт не менять —
  только применять практики в целевом проекте.

## Отчёт

1. Показать применённый чек-лист: какие практики применены и какими правками.
2. Отдельно перечислить практики, которые прочитаны, но не применены, с причиной
   (неприменимо к стеку/уже сделано/вне scope).
3. Указать, какие изменения требуют ревью или ручного решения пользователя.
