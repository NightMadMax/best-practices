# Оценка и принятие кандидатов

Задача skill — отдельный от импорта шаг: разобрать файлы-кандидаты и решить
судьбу каждого. Импорт (`harvest-practice-candidates`) кандидатов не принимает и
`practices/` не трогает; это делает только этот workflow.

## Подготовить контекст

1. Найти корень базы `Best Practices` (`README.md`, `INDEX.md`, `practices/`,
   `candidates/`).
2. Полностью прочитать `candidates/README.md`, `practices/_TEMPLATE.md`,
   ADR-0003 (схема, lifecycle, evidence levels) и текущие
   файлы релевантных разделов `practices/**/README.md` (`1c`, `web`, `common`,
   `tools`, `anti-patterns`, `prompts`, `snippets`).
3. Взять файлы `candidates/PC-*.md` со `status: new` или `triaged`. Если
   пользователь назвал конкретные `id`, работать только с ними.

## Оценить каждого кандидата

1. **Дедуп:** сверить с существующими практиками в `practices/**`. Если тема уже
   покрыта — не плодить дубль: усилить существующую запись или отклонить с
   пометкой на неё.
2. **Качество:** проверить, что заполнены Контекст, Проблема, Решение и что
   решение конкретно, а не «делай хорошо».
3. **Обобщённость:** урок должен быть полезен минимум двум независимым проектам,
   а не только источнику.
4. **Раздел:** убедиться, что целевой раздел верный (`1c`/`web`/`common` или
   `tools`/`anti-patterns`/`prompts`/`snippets`) и frontmatter `target`
   соответствует.
5. **Чистота:** нет секретов, персональных данных, приватных идентификаторов и
   machine-specific paths. Для `tools`/`snippets` это критично — только способ
   настройки и плейсхолдеры.
6. **Evidence:** проверить уровень: `E1` — один проект, `E2` — два независимых
   проекта или воспроизводимый тест, `E3` — несколько проектов и повторяемая
   автоматическая проверка. Внешняя ссылка сама по себе остаётся `E0`.

## Принять или отклонить

0. **Происхождение обязательно (ADR-0001):** перенести в practice frontmatter
   `source`, `added_by`, `created` и `evidence`. Кандидат без происхождения не
   принимать — вернуть на доработку в harvest.
1. **Accept:** создать ровно один файл из `practices/_TEMPLATE.md` по `target`.
   Basename и `id` должны совпадать с кандидатом. Перенести provenance и
   evidence, заполнить applicability, owner, `last_verified`, `review_by`,
   `supersedes` и `conflicts_with`.
   - `E1` создаёт практику со статусом `trial`.
   - `E2`/`E3` создаёт практику со статусом `accepted`.
   - Для tools в теле обязательны вердикт tech-radar, назначение, настройка без
     секретов и ограничения.
   - Добавить wikilink на практику в README соответствующего раздела.
   - В файле кандидата выставить `status: accepted`, заполнить `decided` (дату)
     и указать commit в теле (`## Notes`).
2. **Reject:**
   - В файле кандидата выставить `status: rejected`, `decided` и записать причину
     в `## Notes` (дубль/частный/недостаточно доказательств/не тот раздел).
   - Файл не удалять — журнал сохраняет историю решений.
3. Не менять чужие уже `accepted`/`rejected` файлы без явной причины.

## Синхронизация

1. Обновить README целевого раздела. Глобальные `INDEX.md`/`docs/README.md` не
   перечисляют каждую практику и меняются только при появлении нового раздела.
2. Запустить `make check`; validator должен подтвердить candidate/practice pair,
   wikilinks, evidence threshold и отсутствие секретов.

## Отчёт

1. Разделить результат на: принято, отклонено, отложено (нужно уточнение).
2. Для каждого принятого указать целевой файл и раздел; для отклонённого —
   причину.
3. Напомнить, что доставка принятых практик в проекты — задача
   `apply-best-practices`.
