# Импорт кандидатов в Best Practices

Задача skill — не переносить уроки в `practices/` напрямую, а найти, очистить и
подготовить кандидатов как **отдельные файлы** `candidates/PC-*.md` (один
кандидат — один файл, чтобы приём от многих контрибьюторов не давал
merge-конфликтов).

Сюда идут технические/стек-уроки (код, конфигурация, типовые ошибки, чек-листы
по 1С/web/общие), а также инструменты и MCP-серверы, анти-паттерны, удачные
промпты и переиспользуемые сниппеты. Мета- и процессные уроки (правила агента,
шаблоны, workflow) сюда не тянуть — см. раздел «Маршрутизация».

## Разделы базы и где искать сигнал

| Раздел базы | Что кладём | Где в проекте искать |
|---|---|---|
| `practices/1c`, `web`, `common` | Стек-уроки, типовые ошибки, чек-листы | DEFECTS, PLAYBOOK, research, ADR, reviews |
| `practices/tools` | Инструменты и MCP-серверы: для чего, вердикт | `TOOLS.md`, манифесты/lock-файлы, `.mcp.json`, `.claude/settings*.json`, `.cursor`/MCP-конфиги |
| `practices/anti-patterns` | Что себя не оправдало | DEFECTS `Won't Fix`, postmortem, «Hold»-решения |
| `practices/prompts` | Удачные промпты и skill-паттерны | `.claude/skills`, `.agents/skills`, промпты в docs |
| `practices/snippets` | Переиспользуемые конфиги/куски кода | повторяющиеся конфиги, шаблоны, scripts |

Для `tools` использовать точные поля validator: `Назначение`, `Вердикт`,
`Настройка`; в настройке **без секретов** указывать только env, файл или флаг.
Для `prompts` использовать `Когда применять`, `Промпт / паттерн`,
`Почему работает`. Никаких токенов, ключей и приватных хостов.

## Найти базу

1. Найти корень базы `Best Practices`: каталог с `README.md`, `INDEX.md`,
   `PROJECT.md`, `practices/` и `candidates/`.
2. Полностью прочитать `candidates/README.md` (схема и статус-модель),
   `candidates/_TEMPLATE.md`,
   `docs/architecture/decisions/ADR-0003-one-practice-per-file.md` и текущие `practices/1c/README.md`,
   `practices/web/README.md`, `practices/common/README.md`. Они имеют приоритет
   над этим workflow.

## Найти проекты-кандидаты

1. Корень рабочего каталога по умолчанию — родитель каталога базы.
2. Просматривать только каталоги, которые реально являются git-репозиториями.
   Не доверять именам папок без проверки.
3. По умолчанию исключать:
   - саму базу `Best Practices`;
   - стандарт `new-project-rules` (`Правила для нового проекта`);
   - временные каталоги, кэши, vendor;
   - `.git`, `.obsidian`, `node_modules`, `.venv` и другие служебные директории;
   - репозитории без локального доступа на чтение.
4. Если пользователь сузил область, соблюдать её буквально: только указанные
   проекты, каталоги или темы.

## Собрать сигналы

1. Искать прежде всего checked-in артефакты:
   - `docs/quality/DEFECTS.md`;
   - `docs/quality/PLAYBOOK.md`;
   - `docs/research/`;
   - ADR и postmortem;
   - `docs/reviews/`;
   - `docs/operations/runbooks/`;
   - `CHANGELOG.md`, если там есть доказуемый reusable lesson.
2. Не считать локальную memory единственным доказательством.
3. Для каждого сигнала отделять:
   - наблюдаемый факт;
   - локальный проектный контекст;
   - возможный общий технический урок.
4. До создания файлов показать пользователю harvest preview: рабочее название,
   целевой section, source/evidence и ожидаемую пользу каждого кандидата.
   Отдельно перечислить отброшенные сигналы с причиной. Не скрывать состав
   harvest до финального отчёта.

## Маршрутизация

Определить природу урока и направить его правильно:

1. **Технический урок или актив базы** (стек-урок, инструмент/MCP,
   анти-паттерн, промпт, сниппет) → кандидат в этот backlog. Определить целевой
   раздел по таблице выше: `1c`/`web`/`common`, либо `tools`, `anti-patterns`,
   `prompts`, `snippets`.
2. **Мета/процессный урок пользователя** (про правила агента, структуру
   документации, шаблоны, workflow, валидаторы) → не отправлять напрямую в
   `new-project-rules`. Нормализовать его как BP-кандидат: обычно `prompts` для
   agent/skill/prompt-паттерна или `common` для общего инженерного workflow.
   После обычного review только администратор может затвердить принятую,
   вызревшую практику в стандарт через maintainer-only маршрут
   `promote-project-knowledge → apply-promotion-candidate`.
3. Сразу отбрасывать кандидаты, которые:
   - зависят от частной инфраструктуры, бизнеса или закрытых данных;
   - нельзя очистить от секретов, private identifiers или machine-specific paths;
   - не выглядят повторяемыми хотя бы для двух независимых проектов;
   - не выражаются как практика формата Контекст/Проблема/Решение/Источник.

## Нормализовать кандидатов

1. Для каждого прошедшего фильтр кандидата создать **отдельный файл**
   файл через `scripts/new_candidate.py` с collision-resistant ID
   `PC-<год>-<12hex>-<slug>.md`. Заполнить frontmatter: `id`, `status`, `source`,
   `added_by`, `stack`, `target`, `evidence_level`, `evidence`, `created`,
   `decided` — и тело.
   - **Происхождение обязательно** (ADR-0001): `source` — проект и артефакт;
     `added_by` — кто предложил/подтвердил (пользователь по имени/роли, автор
     коммита; если найдено автоматически — указать `агент`/`harvest`).
   - Без заполненных `source` и `added_by` файл кандидата не создавать.
2. `target` — будущий отдельный файл практики с тем же basename:
   `practices/<раздел>/PC-<год>-<12hex>-<slug>.md`. Не указывать README.
3. `evidence_level`: `E0`, если есть только мнение/внешний материал; `E1`, если
   сигнал подтверждён одним проектом. Harvest не выставляет `E2`/`E3` без
   явных независимых evidence.
4. `status`: `new`, если раздел/target ещё неясны; `triaged`, если кандидат
   нормализован, раздел и target определены и приватный контекст удалён.
5. Не создавать дубли. Если похожий кандидат уже есть в `candidates/` — обновить
   его файл (`evidence`, тело), а не заводить новый.
6. Не выставлять `accepted` автоматически: принятие — задача
   `review-practice-candidates`.

## Записать кандидатов

1. Создать/обновить файлы в `candidates/`; один PR — один новый файл, чтобы
   избежать merge-конфликтов между контрибьюторами (см. `CONTRIBUTING.md`).
2. Сохранить краткие ссылки на исходные артефакты без копирования приватного
   текста.
3. Не трогать файлы `practices/**` — их меняет только review-этап.
4. Запустить `make check`; не считать кандидат готовым при ошибке validator.

## Отчёт

1. Вернуть shortlist кандидатов с причинами, почему они выглядят
   кросс-проектными и к какому стеку отнесены.
2. Явно разделить:
   - какие файлы кандидатов созданы/обновлены в `candidates/`;
   - что отфильтровано и почему;
   - какие meta/process-кандидаты созданы в BP и потенциально пригодны для
     будущего maintainer-only затвердевания в `new-project-rules`;
   - какие кандидаты готовы к `review-practice-candidates`.
3. Не заявлять, что практика перенесена, пока её не принял review-этап.
